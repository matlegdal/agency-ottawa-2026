Project Overview

This repository brings together four major Canadian government open data sources into a shared PostgreSQL database, with separate schemas to prevent collisions. Each dataset has its own pipeline for downloading, cleansing, importing, verifying, and analyzing data.

Shared Database, Isolated Schemas

All four modules share the same PostgreSQL database on Render. Each uses its own schema (cra, fed, ab, general) so tables never collide. Cross-dataset queries use schema-qualified names like cra.cra_identification or fed.grants_contributions.

The Fuzzy Matching Challenge

The defining challenge across these datasets is entity resolution. The same organization can appear under dozens of name variations across CRA, FED, and AB. For example, "The Boyle Street Service Society" appears as 30+ aliases including trade names, truncations, and typos. The general module provides a multi-layer resolution engine with AI-assisted review.

Quick Start Guide

Prerequisites

Node.js 18+ required. All data is already loaded in the shared database — you only need to install dependencies to start querying and running analysis.

1

Clone and Install

git clone <repo-url> && cd hackathon
for dir in CRA FED AB general; do (cd $dir && npm install); done
2

Verify Database Access

Each module ships with .env.public containing read-only credentials. Test connectivity:

cd CRA && npm run verify
cd ../FED && npm run verify
cd ../AB && npm run verify
3

Configure API Keys (optional, for AI features)

To use AI-assisted entity resolution, add credentials to general/.env.public or your personal general/.env:

# Option A: Anthropic Direct API
ANTHROPIC_API_KEY=sk-ant-api03-your-key-here

# Option B: Vertex AI (Google Cloud)
VERTEX_SERVICE_ACCOUNT_JSON={"type":"service_account",...}
VERTEX_PROJECT_ID=your-project-id
VERTEX_CLAUDE_SONNET_MODEL=claude-sonnet-4-6
VERTEX_ENDPOINT=aiplatform.googleapis.com
VERTEX_LOCATION_ID=global
VERTEX_METHOD=rawPredict

The system tries Anthropic first, falls back to Vertex automatically. You can force a provider with --provider vertex.

4

Run Analysis

# CRA: Detect circular gifting patterns
cd CRA && npm run analyze:all

# AB: Run all 6 advanced analysis scripts
cd ../AB && npm run analyze:all

# General: Resolve an entity across all datasets
cd ../general
node scripts/resolve-entity.js --name "Boyle Street Service" --bn 118814391 --llm
5

Adapt to Your Own LLM

The AI review module (general/lib/llm-review.js) has a clean provider abstraction. To add a new provider:

  1. Add a callYourProvider(prompt, options) function returning { text, usage }
  2. Register it in availableProviders() by checking your env variable
  3. Add it to the fallback chain in callLLM()

The prompt and response parsing are provider-agnostic — any LLM that returns JSON works.

Environment Loading Order

All modules load environment variables consistently:

  1. .env.public loaded first (shared defaults for hackathon participants)
  2. .env loaded second with override: true (personal overrides win)

Participants without a .env get the shared read-only credentials. Developers with a .env containing admin credentials automatically override for write operations (migrations, imports, drops).

5 Sole-Source Scenarios

  1. Repeat vendors — Same vendor, same ministry, 3+ sole-source contracts
  2. Sole-source to competitive pipeline — Vendors in both sole-source AND Blue Book
  3. Contract splitting — Multiple same-FY contracts suggesting threshold avoidance
  4. Geographic concentration — Out-of-province vendor analysis
  5. Duration/value outliers — Longest contracts, highest daily rates

5 Grants/Contracts Scenarios

  1. Grant-to-contract ratio — Which ministries favor grants vs procurement
  2. Recipient concentration (HHI) — Monopoly programs
  3. Funding volatility — Most unstable year-over-year programs
  4. Lottery funding — Lottery vs non-lottery distribution patterns
  5. Payment timing — End-of-fiscal-year spending spikes

Entity Resolution Engine

Multi-layer resolution that handles the challenge of matching entities across CRA, FED, and AB where organizations appear under dozens of name variations.

Resolution Layers

LayerMethodConfidenceDescription
1BN Anchor99%Match on root 9-digit business number (gold standard)
1bBN NegativerejectDifferent BN root = confirmed different entity
2Exact Normalized95%Exact match after normalization
3Core Token Gate70-90%All discriminating words must be present
4Trigram + Gate60-85%pg_trgm similarity filtered by tokens
5Trade-Name ExpansionvariesParse "TRADE NAME OF" and bilingual "|"
6AI ReviewvariesClaude contextual judgment (SAME/RELATED/DIFFERENT)

Commands

# Deterministic resolution (no API key needed)
node scripts/resolve-entity.js --name "Boyle Street Service" --bn 118814391

# With AI second pass (Anthropic -> Vertex fallback)
node scripts/resolve-entity.js --name "Homeward Trust" --bn 834173627 --llm

# Force a specific provider
node scripts/resolve-entity.js --name "mustard seed" --llm --provider vertex

# Save results to JSON
node scripts/resolve-entity.js --name "Homeward Trust" --llm --output results.json

# Basic fuzzy search (simpler, trigram-only)
node scripts/fuzzy-search.js --name "University of Alberta"

# Batch cross-dataset matching
node scripts/cross-match.js --threshold 0.6 --limit 200

# Standalone AI review from saved results
node scripts/llm-review.js --input results.json --output reviewed.json
