Notable design decisions

Six properties of the architecture above that matter once it starts running.
№ 01
Two cooperating agents, walled off
The orchestrator publishes findings, runs the workflow, and narrates each step. The verifier subagent can only run SQL — it cannot publish, cannot list schemas, and cannot promote a refutation it has issued. Different prompt, different tools, same SDK.
№ 02
Skills as auto-loaded markdown
Three SKILL.md files load on description match when a question matches their domain. Methodology, data quirks, and naming conventions live in the repo as markdown — versioned in git, never pasted into a system prompt that goes stale.
№ 03
Belt-and-suspenders SQL safety
The DB role is read-only. The postgres-mcp server runs in restricted mode and refuses destructive SQL at the parser. A PreToolUse hook regex-denies DROP / UPDATE / DELETE / ALTER before the parser ever sees the query. Three independent layers.
№ 04
Glass-walled execution
Every SQL query, skill load, subagent step, and tool call streams over a single websocket to the UI. The agent narrates each step itself between tool calls — there is no dark phase of execution, and the reasoning is observable rather than reconstructed after the fact.
№ 05
Deterministic enumeration
The candidate-enumeration query runs verbatim every run, ordered by a fixed sort key. Same DB state produces the same candidate list and the same ordering — what is shown on stage is reproducible from the database alone, not from the model.
№ 06
Custom in-process MCP for the UI
Three publish channels — universe, finding, dossier — emit structured events the UI consumes. The agent doesn't write HTML or markdown for the dashboard; it publishes typed data and the UI does the rendering. Audit, replay, and snapshot become trivial.

04

Methodology decisions.

Choices baked into the queries, the verifier chain, and the framing of the output. Each one is a working file in the repo.

01
Universe panel
Five gate-drop counts published once per run alongside the candidate list. The audience can audit how many recipients each filter dropped — not just see the survivors.
02
CORP federal corporate registry corroboration
Federal corporate registry status (active / amalgamated / dissolved) used as an independent kill-switch. A temporal gate against BN-reuse prevents flagging a successor entity as the original.
03
PA audited Public Accounts
Public Accounts answers "did the cash actually move?" Bilingual | split for Quebec recipients, recipient_name_norm join key. Independent of the grant database.
04
Iterative-exploration loop
On AMBIGUOUS, the orchestrator gets a 3-query budget to defend, revise, or concede. The card transitions pending → challenged → verified (or refuted) so the reasoning is visible.
05
Ghost Capacity hand-off
A refuted-as-zombie + self-dissolved + live-agreement combination is surfaced as a Challenge №2 lead instead of being discarded. Related anomalies are caught as a byproduct.
06
T3010 filing-window math
CRA gives charities six months after fiscal year end to file. The query computes fpe + 6mo ≤ scrape_date before treating silence as evidence of dissolution.
07
F-3 cumulative-amendment immunity
Naïve SUM(agreement_value) over the federal grants table triple-counts cumulative amendments (~$921B reported vs ~$816B actual). Every query reads through fed.vw_agreement_current instead.
08
BN root vs 15-char BN normalization
LEFT(bn, 9) everywhere with a ^[1-9][0-9]{8}$ guard regex. Drops placeholder values (-, all-zeros, 100000000) that would otherwise group unrelated entities.
09
Audit-leads framing
Every output passes a legal-disclosure smell test. Language scrubbed of "fraud / stole / criminal / misappropriated" — the briefing names a finding without naming a crime.
